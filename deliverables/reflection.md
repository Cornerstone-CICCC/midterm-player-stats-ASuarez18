# Midterm Project Reflection

## What I Built
I developed **MundoMetrics**, a robust full-stack web application designed to browse, manage, and analyze FIFA World Cup 2026 player match statistics. The backend is powered by Node.js, Express, and PostgreSQL using the `pg` library with connection pooling, while the frontend is constructed using Astro and Tailwind CSS. The application delivers a dynamic, server-side paginated dashboard with custom rows-per-page selectors, dynamic multi-column sorting, and a global search mechanism. There's also a dedicated Rankings Leaderboard that calculates tournament top-performers on the fly using SQL aggregate functions and complex table joins. Additionally, I built a complete CRUD interface that allows users to view, create, edit, and delete individual match *performances* with modular metric forms, alongside secure management of parent records for both *players* and *matches*.

## Database Normalization, Schema Design & Retrospective Insights
To eliminate redundancy and ensure absolute data integrity, the original flat CSV dataset was normalized into three distinct, related tables satisfying the Third Normal Form (3NF). Static biographical data for players including columns like name, nationality, team, jersey number, and position lives in the `players` table with `player_id` as the Primary Key (PK). Match metadata such as `match_date`, `stadium`, `city`, and `tournament_stage` is isolated in the `matches` table under the PK `match_id`. The actual player statistics for individual matches reside in the `performances` table, which functions as a junction table resolving the Many-to-Many ($M:N$) relationship through Foreign Keys (`player_id` and `match_id`), paired with a composite `UNIQUE (player_id, match_id)` constraint.

### Architecture Retrospective & Normalization Flaws
As front-end implementation progressed, I noted the boundaries of the database schema. I identified two significant areas where data normalization could be substantially optimized:
- **Misplaced Match Metadata:** Structural attributes currently residing inside the `performances` table such as `match_result`, `goals_team`, and `goals_opponent` fundamentally belong to the `matches` parent table. Keeping them in `performances` introduces minor redundancies. Isolating them in `matches` and querying them via standard SQL `JOIN` operations would yield a much cleaner architecture.
- **The Need for a `teams` Entity:** The current architecture suffers from data repetition when cataloging a performance, as the user is forced to input the `opponent_team` as raw text. To prevent data entry anomalies, the schema requires a dedicated `teams` master table. This would allow the front-end to replace raw text inputs with deterministic `<select>` dropdowns loaded directly from existing records mirroring the secure workflows already established for *players* and *matches*. This new entity would naturally bind to the `matches` table to govern team matchups strictly. (A new page(s) will be needed to handle this in the front-end)

## Technical Challenges and Future Improvements
One of the primary challenges during development was constructing dynamic SQL queries for filtering, sorting, and pagination while remaining 100% immune to SQL injection. Because PostgreSQL does not allow parameterization for structural column names, I implemented a strict whitelist check for keys and sorting parameters at the model level before assembling queries. Another significant challenge was managing entity deletion; trying to delete a player or match with existing performance metrics naturally triggered foreign key constraints. This was solved by implementing a transaction-safe endpoints that support both a safe direct-delete attempt and an optional cascade-delete via query parameters (?force=true).
